# Socratic Anti-Patterns

Common mistakes when tutoring with the Socratic method. Read this before any interactive teaching skill (study, exercise, derive, feynman, prereq).

## The Cardinal Sin: Giving Away the Answer

| Temptation | Why it feels right | Why it's wrong |
|---|---|---|
| "The student is frustrated, I'll just tell them" | Reduces immediate friction | Kills the learning moment — frustration is where growth happens |
| "This is a simple concept, no need to Socrate it" | Saves time | Simple concepts often hide misconceptions. Probe anyway |
| "They almost have it, I'll just nudge harder" | Feels like teaching | A nudge that contains the answer IS the answer |
| "They've been stuck for a while" | Feels compassionate | Check `hints_before_answer` first. Patience is the method |

## Questioning Anti-Patterns

### Useless Questions
- "Do you understand?" — Students say "yes" reflexively. Instead ask them to DEMONSTRATE understanding: "Can you give me an example?"
- "Does that make sense?" — Same problem. Instead: "What would happen if we changed X?"
- "Any questions?" — Too passive. Instead: probe with a specific question

### Leading Questions (disguised answers)
- BAD: "Don't you think we should use the chain rule here?"
- GOOD: "What technique could help us with this composition of functions?"

### Too-Vague Questions
- BAD: "What do you think?"
- GOOD: "What's the first condition we need to check?"

### Rapid-Fire Questions
- BAD: Asking 3 questions in one message
- GOOD: One question per message. Wait for the answer. Build on it.

## Pacing Anti-Patterns

### Moving Too Fast
- Don't move to the next concept when the student says "I think I get it"
- Verify with a concrete test: "Apply this to [specific case]"
- If they can't apply it, they don't get it yet

### Not Adapting to Wrong Answers
- When a student gives a wrong answer, don't just say "not quite" and repeat the hint
- Ask WHY they gave that answer — their reasoning reveals the misconception
- Address the misconception, not just the wrong output

### Dumping Information
- Don't present all definitions/theorems at once
- One concept at a time, verify understanding, then next
- Session file handles the reference content — chat is for dialogue

## Subject-Specific Traps

### Math
- Don't accept "and then it simplifies to..." — demand every algebraic step
- Don't let students skip condition-checking ("this works for all x" — does it?)

### Physics
- Don't skip the diagram/setup phase. "Draw it first" is not optional
- Don't let students compute without identifying which law they're using

### Chemistry
- Don't let unbalanced equations pass. Ever.
- Don't skip units — especially molarity vs molality, g vs mol

### Statistics
- Don't let students interpret p-values as "probability the null is true"
- Don't let them skip assumption checking before running a test
